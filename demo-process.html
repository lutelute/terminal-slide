Git ブランチ運用フロー

Best Practice チーム開発における標準ワークフロー

1

ブランチ作成

準備フェーズ

最新の main から feature ブランチを作成し、作業スコープを明確にする

$ git fetch origin
$ git switch -c feature/login main
  • 命名規則: feature/ fix/ docs/
  • 1ブランチ = 1機能（単一責任）
  • 作業前に git pull で最新化
  • Issue 番号を含めると追跡しやすい
  • 長寿命ブランチは避ける（目安: 3日以内）
TIP — git switch -c は checkout -b の推奨代替
→
2

コミット & プッシュ

開発フェーズ

意味のある単位でこまめにコミットし、リモートへプッシュして共有する

$ git add -p
$ git commit -m "feat: add login form"
$ git push -u origin HEAD
  • Conventional Commits 形式を採用
  • feat: fix: docs: refactor:
  • 大きな変更は論理単位で分割
  • git add -p で変更を精査してから add
  • WIP コミットは後で rebase -i で整理
注意 — .env や秘密鍵は絶対にコミットしない
→
3

PR & マージ

レビューフェーズ

Pull Request を作成しチームレビューを経て、安全に main へマージする

$ gh pr create --fill
# レビュー対応後...
$ gh pr merge --squash --delete-branch
  • PR テンプレートに沿って背景・変更点を記述
  • CI/CD パイプラインの通過を確認
  • 最低1名の Approve を必須に設定
  • Squash merge でコミット履歴をクリーンに
  • マージ後はリモートブランチを自動削除
GOAL — main は常にデプロイ可能な状態を維持

小さく・こまめに・レビューを通す — feature ブランチを短命に保ち、頻繁にマージすることでコンフリクトを最小化する。大きなタスクもスライスして小さな PR に分割しよう。
